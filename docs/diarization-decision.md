# 说话人分离需求分析与当前决策

## 1. 结论

当前阶段暂不实现说话人分离，直接推进 F5 文本规整。

原因：

- 当前 ASR 已能产出可读转写稿，满足进入文本规整的前置条件。
- 当前目标是会议记录规整和内容提要，不强依赖“谁说了什么”。
- 说话人分离会引入新的模型、依赖、许可证和离线部署复杂度。
- 说话人标签一旦错误，会污染转写稿可读性和后续规整结果。

## 2. 底层原理差异

当前不分离方案关注“说了什么”：

```text
音频 -> ASR -> 带时间戳文本
```

说话人分离关注“什么时候是谁在说话”：

```text
音频
  -> VAD 语音活动检测
  -> 说话人嵌入 speaker embedding
  -> 聚类 clustering
  -> 时间段归属 Speaker 1 / Speaker 2
  -> 与 ASR 时间戳对齐
  -> 带说话人标签的转写稿
```

两者不是同一个任务。ASR 识别语言内容，diarization 识别声纹/音色/说话模式并做时间聚类。

## 3. 什么场景需要说话人分离

需要优先考虑说话人分离的场景：

- 需要还原“谁说了什么”。
- 会议中存在争议，需要保留发言归属。
- 后续要抽取负责人、行动项、承诺事项。
- 需要区分主持人、客户、供应商、内部团队。
- 需要按人统计发言或生成个人维度纪要。
- 合规或审计要求保留发言人归属。

可以暂不做说话人分离的场景：

- 只需要会议整体内容提要。
- 只需要规整原始转写稿，提高可读性。
- 不抽取负责人和行动项。
- 会议参与人数少，或发言归属可以人工判断。
- 说话人分离准确率不稳定，错误标签反而影响阅读。

## 4. 当前项目判断

当前聚焦功能是：

- F1 音频上传。
- F3 语音识别。
- F5 文本规整。
- F6 会议提要。

因此当前可以接受：

```text
音频 -> ASR 转写稿（带时间戳） -> 文本规整 -> 内容提要
```

暂不要求：

```text
音频 -> Speaker N 归属 -> 按人组织会议内容
```

## 5. 保留的扩展点

虽然当前跳过 F4，但保留以下扩展点：

- 配置项继续保留 `enable_diarization`。
- 转写稿格式继续兼容 `Speaker N：文本`。
- `ASRSegment` 数据结构保留 `speaker` 字段。
- 后续如果需要，可以新增独立 provider，例如 pyannote、FunASR 或 NeMo。

## 6. 后续重新评估条件

当出现以下需求时，再重新评估说话人分离：

- 用户明确需要“谁提出了哪个观点”。
- 进入会议结论、行动项、负责人抽取阶段。
- 会议参与方复杂，例如客户、供应商、内部多部门混合。
- 需要审计或追责。
- 用户愿意提供参会人名单或声纹样本。
